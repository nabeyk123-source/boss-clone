## 1. コンセプト

### 1.1 「わたなべ部長との対話感」

Web UI のコンセプトは「**信頼できる上司と対話している体感**」。

- ビジネスチャット風（Slack / Teams 的な信頼感）
- ただし冷たくない、温かみがある
- 圧迫感のない明るめのトーン
- ストレスフリーなUX

### 1.2 デザイン原則

1. **シンプル優先**：機能を盛り込みすぎない
2. **わたなべ部長らしさ**：装飾より、対話の質で勝負
3. **マルチエージェントの中身は隠す**（ただし思考プロセスタブで意図的公開）
4. **デモ映え**：審査員が画面を見て1秒で理解できる構造

#### ユーザー発言バブル

- 配置：**中央**

### 3.3 質問選択肢ボタンの設計

```
Q1: 重視したい観点は？（複数選択可）

[スピード]  [品質]  [コスト]  [チーム負荷]  [その他...]
```

**ボタン仕様**：
- 角丸：20px（カプセル型）
- 通常：白背景、青ボーダー（`#1976D2`）、青文字
- ホバー：薄い青背景（`#BBDEFB`）
- 選択中：濃い青背景（`#1976D2`）、白文字
- 「その他...」をクリック：テキスト入力欄が下に展開

**複数選択時**：
- クリックで選択/解除トグル
- 選択中は色変化で明示
- 「回答する」ボタンで確定

**単一選択時**（System1の `allow_multiple: false`）：
- クリックで即確定 + Synthesizer起動

### 3.4 思考中の表現

ユーザーが質問に回答 → Synthesizer 起動中の表示：

```
わたなべ部長が考え中...
⏺ ⏺ ⏺
```

または：

```
わたなべ部長が論点を整理中...
（裏側で熟考エージェントの結果待ち）
```

**動的にメッセージ変更**：
- 0-5秒：「考え中...」
- 5-15秒：「論点を整理中...」
- 15秒以降：「もう少しで回答できる...」

これで「ただ待つ」感を緩和。

### 3.5 メッセージ入力欄

- 画面下部固定
- プレースホルダー：「相談を入力...」
- Enter で送信、Shift+Enter で改行
- 送信ボタン：青の円形（`#1976D2`）に矢印アイコン

### 4.3 構成要素

#### エージェントカード（System1、System2 横並び）
- 上部：エージェント名 + モデル情報
- 中央：完了時間、結論サマリ
- 下部：「詳細を展開」ボタン → クリックで raw_response 表示

#### Synthesizerカード（フル幅）
- メタ情報 + alignment 状態
- 「出力を展開」で最終回答全文表示

#### 並列実行効果の説明
- 静的テキスト
- 逐次 vs 並列の数値比較
- 「**マルチエージェントの必然性**」のアピール

## 9. マルチエージェントロジックとの統合

### 9.1 既存実装の流用

`boss_clone_lib` 配下の以下をそのまま使う：
- `coordinator.py`: ADK Coordinator
- `agents/system1_agent.py`
- `agents/system2_agent.py`
- `agents/synthesizer_agent.py`
- `retrieval/service.py`
- `session/history.py`

### 9.2 Streamlit側で必要な追加実装

1. **session_state による対話状態管理**
   - 現在のターン（Turn 1 / Turn 2 / 完了）
   - System1 の質問リスト
   - ユーザーの選択肢回答
   - System2 のバックグラウンドタスク
   - Synthesizer の最終回答

2. **非同期処理のラップ**
   - Streamlit は同期実行モデルが基本
   - System2 のバックグラウンド実行は `asyncio` で管理
   - `st.empty()` で「考え中」表示を更新

3. **思考プロセスログの蓄積**
   - 各エージェントの完了時刻、所要時間
   - raw_response の保持
   - 思考プロセスタブで参照

### 9.3 Session State 構造

```python
st.session_state = {
    "current_session_id": str,
    "messages": [
        {
            "role": "user" | "assistant",
            "content": str,
            "timestamp": datetime,
        },
        ...
    ],
    "current_turn": "input" | "answering_questions" | "synthesizing" | "done",
    "system1_output": dict,
    "system2_task": asyncio.Task,  # バックグラウンド実行中
    "system2_output": dict,
    "user_answers": list,
    "synthesizer_output": dict,
    "agent_traces": [  # 思考プロセスタブ用
        {
            "agent": "system1" | "system2" | "synthesizer",
            "started_at": datetime,
            "completed_at": datetime,
            "duration_ms": int,
            "raw_response": str,
        },
        ...
    ],
}
```

## 11. Cloud Run デプロイへの引き継ぎ

### 11.1 Streamlit を Cloud Run にデプロイする手順（Day 5以降）

1. **Dockerfile 作成**
   ```dockerfile
   FROM python:3.11-slim
   WORKDIR /app
   COPY requirements.txt .
   RUN pip install -r requirements.txt
   COPY . .
   CMD streamlit run scripts/boss_clone_web.py --server.port=8080 --server.address=0.0.0.0
   ```

2. **gcloud デプロイ**
   ```bash
   gcloud run deploy boss-clone-web \
     --source . \
     --region us-central1 \
     --allow-unauthenticated \
     --memory 1Gi
   ```

3. **環境変数 / シークレット**
   - Vertex AI 認証は Cloud Run のサービスアカウントで自動付与

## 13. リスクと対策

| リスク | 影響 | 対策 |
|---|---|---|
| Streamlit と asyncio の相性問題 | バックグラウンド実行の挙動不安定 | Phase 2-1 早期にテスト、別パターン用意 |
| カスタムCSSが効きにくい | 「部長対話感」が出ない | Phase 2-5 で集中調整 |
| Cloud Run デプロイ時の認証問題 | デモURL公開できない | Phase 2 後の別タスクで早期検証 |
| アイコン画像未提供 | デモ品質に影響 | プレースホルダーで開始、後で差し替え |
