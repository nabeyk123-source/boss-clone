# デモシナリオ集

**Version**: 1.0.0
**作成日**: 2026-06-23
**対象**: ハッカソン提出物のデモンストレーション
**前提**: マルチターン CLI（`python scripts/boss_clone_chat.py`）が動く状態

---

## デモ前提と表記

すべてのシナリオは **わたなべ部長クローン** との対話を、マルチターン CLI 上で実演する形式。

```
あなた> [相談内容]
    ↓ (Turn 1: 8〜15秒で表示)
🧠 System1 の直感: [結論+気になる点]
🙋 確認させてください: [質問 2〜3個]
    ↓ (ユーザーが回答する 10〜30秒の間に System2 がバックグラウンドで完走)
あなた> [質問への回答]
    ↓ (Turn 2: System2 await + Synthesizer = 16〜24秒)
🎯 わたなべ部長として: [統合判断（出力にはシステム実装が漏れない）]
```

体感待ち時間 = Turn 1 + Turn 2 = **約 25〜37 秒**（ユーザー回答時間は「待ち」じゃない）。

---

## シナリオ A: 一致パターン（軽量デモ、約 30 秒）

### 用途
ハッカソン審査の **アイスブレイク**。「動くデモ」をまず見せて、システム全体像を伝える。

### 想定背景
わたなべ部長クローンの「素直な確信」が出るパターン。直感・熟考が両方「やった方がいい」で揃うため、サクッと結論が出る。

### ユーザー入力
```
新サービスにセキュリティレビューが必要かどうか判断したい
```

### Turn 1 で表示される System1 の質問（実測例）
1. 新サービスは、顧客の個人情報や機密情報を扱いますか？
2. リリース予定日はいつで、セキュリティレビューに割ける期間はどのくらいですか？
3. 過去に類似サービスでセキュリティインシデントは発生していますか？

### ユーザー回答例（デモ用、簡潔に）
```
1. 顧客の決済データを扱う、本人確認情報も含む
2. リリース予定は3ヶ月後
3. 過去のインシデントなし
```

### Turn 2 で表示される統合判断（要点）
- 論点1: リスクの大きさの特定 — 「これは MUST だ」と即断
- 論点2: レビュー範囲の明確化 — 自社開発 vs 外部 SaaS 連携の線引き
- 論点3: スピードとコストの現実的な計画 — トレードオフ提示

**結論**: 一致パターンなので「方向は見えている、あとは詰めるだけ」のテンポで進む。

### デモでのアピールポイント
- 「相談 → 質問 → 答える → 統合判断」のリズムが対話的
- Turn 1 で 8秒で質問が返ってくる「即決感」
- 統合判断にシステム実装の語彙（直感/熟考/エージェント）が一切出ない自然さ
- MUST/WANT 分離が自然に出る（わたなべ部長スタイル）

## シナリオ B: 不一致パターン（メインデモ、約 1 分、★最重要）

### 用途
**マルチエージェントの必然性が一番伝わる**。System1（直感）と System2（熟考）の結論が割れて、Synthesizer がそれを噛み砕いて統合する瞬間を見せる。

### 想定背景
営業の現場感（過去 OK パターン → 採用）と、戦略・規程の網羅検討（KPI 整合性・リソース・実行計画の懸念 → 保留）が衝突する典型ケース。

### ユーザー入力
```
来月、kabe-anon の新機能をリリースしたいです
```

### Turn 1 で表示される System1 の質問（実測例）
1. リリースする新機能の具体的な名称と、その機能が解決する課題は何ですか？
2. リリース後の目標とする KPI（例：利用率、売上貢献）と、その達成見込みはどの程度ですか？

### ユーザー回答例
```
1. kabe v3 の新規ダッシュボード機能、ユーザー数を 15% 拡大したい
2. リリース日は来月25日、社内レビューは来週完了予定
```

### Turn 2 で表示される統合判断（実測例、抜粋）

```
OK、kabe-anonの新機能リリースの件、少し整理しようか。
やりたい気持ちはよく分かった。方向性としては面白そうだし、
前に進めたい。ただ、いくつか論点があるから、一度立ち止まって考えたい。

論点1: なぜ「ユーザー数15%増」につながるのか？
- 状況: 新しいダッシュボードでユーザー体験が良くなるのは分かる。でも、
  それがどう「新規ユーザー獲得」や「既存ユーザーの拡大」に繋がるのか、
  そのストーリーがまだ見えない。ここを語れないと、ただの自己満足な
  機能追加で終わってしまう。
- 確認すべき質問: この新機能によって、具体的に誰のどんな行動が変わり、
  結果としてユーザー数が15%増えるというロジックを説明できる？

論点2: この機能の「顧客」は誰か？
- ...

論点3: 「来月25日リリース」の実現性
- ...

まずはこの3点を整理して、もう一度話そうか。期待してるよ。
```

**裏側の動き（デモでは口頭説明）**:
- System1 = 採用（過去の kabe-anon 関連は前向き判断が多い、distance 0.74）
- System2 = 条件付き保留（KPI 整合性・実行計画の論点 2 件抽出）
- Synthesizer がこの不一致を「やりたい気持ちは分かる、ただ立ち止まる」と自分の言葉に統合

### デモでのアピールポイント
- **マルチエージェントの必然性**: 単発 LLM では「直感 vs 熟考」を独立に走らせる構造的根拠が無い
- **不一致時の処理**: 「直感は採用、熟考は保留」と内部状態を晒さず、わたなべ部長として 1 人の判断に統合
- **問い返し型**: すぐ答えず「もう一度話そうか」と部下の思考を深める
- **MUST/WANT 分離・トレードオフ明示**: わたなべ部長スタイル 5 特徴が自然に出る

## シナリオ C: 複雑判断パターン（深いデモ、約 2 分）

### 用途
**戦略性と人間関係の両方を扱える** ことを示す。顧客との信頼関係・社内リソース・事業戦略が複雑に絡む相談。

### 想定背景
営業企画部の典型的な悩み — 「戦略顧客の追加要望に応えるか、社内リソースの限界を取るか」。直感は過去 NG パターン（顧客信頼を損なう）、熟考は条件付き保留（要件次第）。

### ユーザー入力
```
顧客の追加要望を一旦保留にしたいけど大丈夫？
```

### Turn 1 で表示される System1 の質問（実測例）
1. 保留にすることで、顧客からの信頼失墜リスクはどの程度あるか？
2. 保留期間中、顧客への代替案や進捗報告の計画はあるか？

### ユーザー回答例
```
1. 戦略顧客の A 社、契約金額は年 3000万、要望は分析ダッシュボードの追加
2. 工数 40 人日見込み、代替提案として既存 Lite 機能の活用を提示済み
```

### Turn 2 で表示される統合判断（実測例、抜粋）

```
お疲れさん。相談の件、少し整理しようか。
君からの情報で輪郭は見えてきたけど、判断するにはまだ早い。
ここからが大事なところだ。論点は3つある。

論点1: 「信頼失墜リスク」の解像度
- 状況: 今のところ「戦略顧客のA社」という事実だけで、リスクの具体的な
  中身が見えてない。「信頼を失う」というのは、具体的に何を失うことか？
  契約解除まであり得るのか、それとも次の発注が減る程度なのか。
  この見立てで、打ち手は大きく変わる。
- 確認すべき質問: この要望を保留にした場合、A社との関係で最悪
  どんな事態が想定される？先方の担当者の温度感は？

論点2: 「保留」か「計画的延期」か
- 状況: 単に「保留したい」だと、ただの先送りだ。でも、「短期的な
  開発リソースの確保」を取る代わりに、「顧客への丁寧な説明と
  代替案の提示」という別の仕事をする、と考えれば戦略的な判断になる。
  このトレードオフを意識できているかがポイントだ。
- ...

論点3: この要望の「戦略上の位置づけ」
- 状況: この「分析ダッシュボード」は、我々の事業戦略
  （データ活用推進など）から見て、絶対に必要な「MUST」な機能か、
  それともあると嬉しい「WANT」な機能か。
- ...

この3つを整理して、もう一回持ってきてくれ。
そうすれば、どう動くべきか決められる。
```

**裏側の動き（デモでは口頭説明）**:
- System1 = 却下（過去に「顧客保留 → 信頼低下」NG パターンを引いた、distance 0.72）
- System2 = 条件付き保留（リソース vs 顧客信頼のトレードオフ、戦略整合性で論点 2 件）
- Synthesizer がユーザー回答（A 社・年3000万・40人日）を踏まえて、論点1 で「具体的に何を失う？」と踏み込む

### デモでのアピールポイント
- **ユーザー回答の反映**: 「A 社」「年 3000 万」「40 人日」がそのまま論点の前提として使われる
- **「保留」の語彙を別案へ転換**: 単なる先送りではなく「計画的延期」という戦略的判断へ
- **再質問の抑制**: System1 が出した質問（信頼失墜リスク、代替案）に既に答えがあるので、Synthesizer は別角度（解像度上げ、戦略上位置づけ）を問う
- **わたなべ部長の口調**: 「お疲れさん」「ここからが大事なところだ」「持ってきてくれ」の温度感

## デモ実演の段取り（推奨）

| 順 | シナリオ | 所要 | 目的 |
|---|---|---|---|
| 1 | A (軽量) | 30 秒 | 「動くデモ」を見せる、リズム伝達 |
| 2 | **B (メイン)** | 1 分 | **マルチエージェントの必然性を伝える** |
| 3 | C (深堀り) | 2 分 | 複雑判断・人間関係を扱えること |

合計 **3.5 分**で 3 パターン見せられる。

### 口頭ナレーション例（シナリオ B 用）

> ここで一番のポイントは、私が「来月リリースしたい」と相談した時、
> システムの裏では 2 体のエージェントが**違う角度**で並行に考えてます。
> 1 体は過去判断パターン（直感）、もう 1 体は規程・戦略・暗黙知の網羅検討（熟考）。
> この 2 体が**違う結論**を出した時、3 体目の統合エージェントが
> 「やりたい気持ちは分かる、ただ立ち止まる」と 1 人の部長として
> 噛み砕いてくれる。これがマルチエージェントである必然性です。

## 補足：体感速度の説明

審査員から「待ち時間長くない？」と聞かれた時の説明：

> このシステムは**ただ待たせない**設計を入れています。
> 相談を投げると 8〜15 秒で確認質問が返り、ユーザーが答えている
> 10〜30 秒の間に「熟考」エージェントが裏で並列に完走します。
> Turn 2 では追加待ち時間が最小化されるので、**体感では対話のリズム**が成立します。
> 単発 LLM が 60 秒沈黙する設計とは、ユーザー体験が根本的に違います。
